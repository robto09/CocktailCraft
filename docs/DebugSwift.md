# DebugSwift Guide for CocktailCraft

## Overview

This guide documents the DebugSwift integration in the CocktailCraft iOS app: installation, the compatibility issues encountered and their solutions, day-to-day usage, and QA testing scenarios. DebugSwift is a comprehensive debugging toolkit for QA and development purposes. The integration is **DEBUG-only** and is not included in release builds.

## Requirements & Compatibility

### DebugSwift Requirements
- **iOS**: 14.0+
- **Swift**: 6.0+ (according to documentation)
- **Xcode**: 16.0+

### CocktailCraft Project Status
- **iOS Target**: 17.0 ✅
- **Swift Version**: 5.10 (see [Issues Encountered & Solutions](#issues-encountered--solutions))
- **Xcode**: 16.0+ ✅
- **Architecture**: Kotlin Multiplatform with shared ViewModels

### Compatibility Challenges Identified
1. Swift version mismatch (project settings vs 6.0 requirement)
2. Kotlin Multiplatform shared code concurrency model conflicts
3. CocoaPods version availability issues
4. Swift 6 strict concurrency checking incompatibility

## Installation

> The Xcode project is generated by XcodeGen (IO-8): `iosApp/project.yml` is the
> source of truth for all build settings. Never edit `project.pbxproj` by hand —
> change `project.yml`, then run `xcodegen generate && pod install`.

### Step 1: Swift Version

The Swift version is set in `iosApp/project.yml` (line 36 for the app target, mirrored on the widget and test targets):

```yaml
settings:
  SWIFT_VERSION: "5.10"
```

The project initially attempted Swift 6.0 to satisfy DebugSwift's stated requirement; it was downgraded to 5.10 for KMP compatibility (see below).

### Step 2: CocoaPods Integration
**Issue**: DebugSwift 2.0.0 not available in CocoaPods trunk

**Initial Attempt**:
```ruby
pod 'DebugSwift', '~> 2.0', :configurations => ['Debug']
```
**Error**: `None of your spec sources contain a spec satisfying the dependency`

**Solution**: Used GitHub source with specific tag
```ruby
pod 'DebugSwift', :git => 'https://github.com/DebugSwift/DebugSwift.git', :tag => '1.7.13.1', :configurations => ['Debug']
```

### Step 3: App Integration
DebugSwift is set up in `CocktailCraftApp.swift`:

```swift
#if DEBUG
import DebugSwift
#endif

@main
struct CocktailCraftApp: App {
    #if DEBUG
    static var debugSwift = DebugSwift()
    #endif

    init() {
        // Initialize Koin
        KoinInitializer.instance.initialize()

        // Register background tasks synchronously: BGTaskScheduler requires
        // all launch handlers to be registered before the app finishes
        // launching.
        BackgroundSyncManager.shared.registerBackgroundTasks()

        // Keep the home-screen widgets' favorites snapshot current
        WidgetDataBridge.shared.start()

        // Initialize DebugSwift for QA debugging
        #if DEBUG
        setupDebugSwift()
        #endif
    }
}
```

## Issues Encountered & Solutions

### Issue 1: Swift 6 Concurrency Errors in DebugSwift
**Error Messages**:
```
Let 'SynchronizableAny' is not concurrency-safe because non-'Sendable' type 'CFString' may have shared mutable state
```

**Root Cause**: DebugSwift library not fully compatible with Swift 6 strict concurrency

**Solution**: The Podfile's `post_install` hook overrides the Swift settings for the DebugSwift pod only (`iosApp/Podfile`):
```ruby
post_install do |installer|
  installer.pods_project.targets.each do |target|
    target.build_configurations.each do |config|
      # Fix Swift 6 concurrency issues for DebugSwift
      if target.name == 'DebugSwift'
        config.build_settings['SWIFT_STRICT_CONCURRENCY'] = 'minimal'
        config.build_settings['SWIFT_VERSION'] = '5.0'
      end
    end
  end
end
```

### Issue 2: Swift 6 Concurrency Errors in Main App
**Error Messages**:
```
Static property 'instance' is not concurrency-safe because non-'Sendable' type 'KoinInitializer' may have shared mutable state
Cannot access property 'sharedViewModel' with a non-sendable type 'SharedCocktailDetailViewModel' from nonisolated deinit
Sending 'self.sharedViewModel' risks causing data races
```

**Root Cause**: Kotlin Multiplatform shared ViewModels not compatible with Swift 6 strict concurrency

**Attempted Solutions**:
1. `SWIFT_STRICT_CONCURRENCY = minimal` - ❌ Still had errors
2. `SWIFT_STRICT_CONCURRENCY = complete` - ❌ Made it worse
3. Removing concurrency settings - ❌ Still had errors

**Final Solution**: Downgraded Swift version to 5.10 (in `project.yml`)
```yaml
SWIFT_VERSION: "5.10"
```

**Why This Works**:
- Swift 5.10 doesn't enforce strict concurrency checking
- Maintains modern Swift features
- Perfect compatibility with Kotlin Multiplatform shared code
- DebugSwift works flawlessly

## Final Configuration

### Project Settings (`iosApp/project.yml`)
```yaml
options:
  deploymentTarget:
    iOS: "17.0"
targets:
  CocktailCraft:
    settings:
      SWIFT_VERSION: "5.10"
      IPHONEOS_DEPLOYMENT_TARGET: "17.0"
```

After any `project.yml` change, regenerate: `xcodegen generate && pod install`.

### Podfile
```ruby
platform :ios, '17.0'

target 'CocktailCraft' do
  use_frameworks!

  pod 'shared', :path => '../shared'
  pod 'Kingfisher', '~> 7.0'
  pod 'DebugSwift', :git => 'https://github.com/DebugSwift/DebugSwift.git', :tag => '1.7.13.1', :configurations => ['Debug']
end

post_install do |installer|
  installer.pods_project.targets.each do |target|
    target.build_configurations.each do |config|
      config.build_settings['ENABLE_PREVIEWS'] = 'NO'
      config.build_settings['SWIFT_EMIT_LOC_STRINGS'] = 'NO'
      config.build_settings['CLANG_ENABLE_MODULES'] = 'YES'

      # Fix Swift 6 concurrency issues for DebugSwift
      if target.name == 'DebugSwift'
        config.build_settings['SWIFT_STRICT_CONCURRENCY'] = 'minimal'
        config.build_settings['SWIFT_VERSION'] = '5.0'
      end
    end
  end
end
```

## Usage Guide

### Accessing DebugSwift
1. **Automatic**: Appears automatically in DEBUG builds
2. **Shake Gesture**: Shake device to toggle interface
3. **Manual**: Call `CocktailCraftApp.debugSwift.toggle()`

### Features Available
- **Network Inspector**: Monitor TheCocktailDB API calls
- **Performance Monitoring**: CPU, memory, FPS tracking
- **App Tools**: Crash reports, console logs, device info
- **Interface Tools**: View hierarchy, touch indicators
- **Resources**: File browser, UserDefaults viewer

### Custom Debug Actions
Access via: **App Tab → Custom Actions → CocktailCraft Debug Tools**

- **Clear Favorites Cache** - removes all saved favorite cocktails
- **Clear Cart Data** - empties the shopping cart
- **Reset App State** - clears favorites, cart, and search history
- **Force Network Error** - simulates network error conditions

## QA Testing Scenarios

### Testing Network Connectivity
1. Open **Network tab**
2. Navigate through app (search cocktails, view details)
3. Monitor API calls to TheCocktailDB
4. Check response times and status codes
5. Verify JSON data structure

### Testing Performance
1. Open **Performance tab**
2. Navigate through different screens
3. Monitor CPU and memory usage
4. Check for memory leaks
5. Verify smooth FPS (60fps target)

### Testing Error Handling
1. Use **Force Network Error** action
2. Navigate through app features
3. Verify error messages display correctly
4. Check app doesn't crash

### Testing Data Persistence
1. Add items to favorites and cart
2. Use **Clear Favorites Cache** or **Clear Cart Data**
3. Verify app handles empty states correctly
4. Test data restoration

### Testing UI Components
1. Open **Interface tab**
2. Use **View Hierarchy** to inspect UI structure
3. Enable **Touch Indicators** to verify touch targets
4. Use **View Borders** to check layout issues

## Troubleshooting

### Build Errors
- **Concurrency Errors**: Ensure Swift 5.10 is used for the main app (set in `project.yml`, then `xcodegen generate && pod install`)
- **DebugSwift Not Found**: Verify CocoaPods installation and GitHub source
- **Code Signing**: Normal iOS requirement, unrelated to DebugSwift

### Runtime Issues
- **Interface Not Appearing**: Check DEBUG build configuration
- **Shake Not Working**: Verify device supports motion events; use manual toggle `CocktailCraftApp.debugSwift.toggle()`
- **Network Monitoring**: Ensure API calls are made to configured URLs (only `https://www.thecocktaildb.com` is monitored)
- **Performance Impact**: DebugSwift itself uses resources; this is normal in DEBUG builds

## Key Learnings

1. **Swift 6 Compatibility**: Not ready for Kotlin Multiplatform projects
2. **CocoaPods vs GitHub**: Use GitHub source for latest versions
3. **Concurrency Models**: KMP and Swift 6 concurrency don't mix well yet
4. **DEBUG-Only Integration**: Critical for production safety
5. **Version Compatibility**: Sometimes older Swift versions are more stable
